# IMPLEMENTATION CHANGELOG ‚Äî PHASE 2
**Chettinad Care ‚Äî Patient Pipeline + Persistence Pass**
**Date:** 2026-04-08

## Summary

Phase 2 completes the patient identity pipeline, replaces all in-memory storage
with database-backed persistence, and closes the remaining security gaps from
the Phase 1 hardening audit.

---

## 1. Schema Changes ‚Äî `backend/database.js`

### New Tables (both SQLite and PostgreSQL)

**`clinical_drafts`**
```sql
key TEXT PRIMARY KEY
data TEXT NOT NULL        -- JSON-serialised draft content
etag TEXT NOT NULL        -- ETag for OCC
updated_at TIMESTAMP
```
Replaces the in-memory `Map()` in `drafts.js`. Drafts now survive server
restarts, container recycles, and rolling updates. Data loss risk eliminated.

**`notifications`**
```sql
id SERIAL / AUTOINCREMENT PRIMARY KEY
type TEXT                 -- 'critical' | 'info' | 'appointment' | 'lab'
title TEXT
body TEXT
patient_id TEXT           -- optional context
actor_id TEXT             -- who triggered the event
target_role TEXT          -- NULL = all staff, or specific role
read INTEGER DEFAULT 0
created_at TIMESTAMP
```
Replaces the hardcoded three-item array in `notifications.js`. Notifications
are now real clinical events generated by the system.

### New Indexes
- `idx_notifications_read` ‚Äî fast unread count queries
- `idx_notifications_created_at` ‚Äî ordered listing
- `idx_users_patient_id` ‚Äî patient portal scope resolution

## 2. Persistent Drafts ‚Äî `backend/routes/drafts.js`

**Before:** `const drafts = new Map()` ‚Äî lost on any container restart.

**After:** Full DB-backed implementation with:
- `GET /:key` ‚Äî SELECT from `clinical_drafts`
- `PUT /:key` ‚Äî SELECT then INSERT or UPDATE with ETag OCC (412 on mismatch)
- `DELETE /:key` ‚Äî DELETE from `clinical_drafts`

OCC logic is identical to the previous implementation ‚Äî If-Match header
enforcement, ETag generation ‚Äî just persisted to DB instead of memory.

**Impact:** Eliminates the highest clinical data-loss risk in the platform.

## 3. DB-backed Notifications ‚Äî `backend/routes/notifications.js`

**Before:** Hardcoded `let notifications = [...]` array with stale `pat-1` references.

**After:**
- `GET /` ‚Äî SELECT from `notifications` filtered by `target_role`, newest first, limit 50

## 5. Security Hardening

### Break-glass minimum justification
**Before:** 10 characters (insufficient ‚Äî "Emergency" is 9 characters)
**After:** 50 characters in both `encounters.js` and `patients.js`

The frontend `PatientDossier.tsx` displays the 50-character minimum in the modal
text. The `BREAK_GLASS_MIN_LENGTH = 50` constant is defined at the top of each
route for easy adjustment.

### Login rate limiting ‚Äî `backend/routes/auth.js`
- Max 10 failed attempts per IP per 15-minute window
- Returns 429 with retry-after information in the message
- Successful login clears the rate limit record for that IP
- In-memory implementation (no additional dependency)
- Resets on server restart ‚Äî acceptable for pilot scale

### Activation rate limiting ‚Äî `backend/routes/activation.js`
- Max 5 failed claim attempts per UHID per 20 minutes
- Tied to patient_id (not IP) ‚Äî prevents OTP brute-force against a specific patient
- Returns 429 with retry-after minutes
